## 1. 项目概述

### 1.1 背景

个人长期项目（学习、副业、健康习惯、年度目标等）往往跨越数月甚至数年。现有工具要么偏短期任务管理（Todo），要么偏笔记记录，缺少对**长期项目结构**、**习惯周期**、**时间投入统计**和**周期性复盘报告**的一体化支持。

本项目旨在构建一款**个人维护、跨全平台**的项目管理 App，核心能力包括：

- 长期项目全生命周期管理
- 任务式与习惯式两种项目模型
- 灵活的时间记录（实时计时 / 事后补录）
- 局域网无主同步（同 Wi‑Fi 下手动触发，无需第三方服务器）
- （V1.1+）周报、年报自动生成
- （可选）与思源笔记 API 联动，将报告与进度写回笔记块

### 1.2 产品愿景

> 用一套数据，在任意设备上管理长期目标；用时间记录证明投入；用报告回顾成长；可选地将精华同步到思源笔记，形成个人知识闭环。

### 1.3 目标用户

- **主要用户**：开发者本人（单用户、个人维护场景）
- **使用特征**：多设备切换（Android / iOS 手机 + Win / Mac 桌面）、重视数据主权与本地优先、希望 AI 辅助开发降低维护成本

### 1.4 V1 成功标准

| 指标 | 目标 |
|------|------|
| 平台覆盖 | Android、macOS、iOS、Windows 四端可用（鸿蒙不在 V1 范围） |
| 核心流程 | 创建项目 → 添加任务/习惯 → 设置 Milestone → 记录时间 → 局域网手动同步 |
| 数据安全 | 本地可完整离线使用；同步不依赖公网与第三方服务 |
| 维护成本 | Tauri 2 单前端 + Rust 后端，个人可独立迭代 |
| 报告 / 思源 | V1 不做；架构预留，V1.1 起逐步实现 |

## 2. 术语定义

| 术语 | 定义 |
|------|------|
| **项目（Project）** | 顶层容器，分为「任务式」或「习惯式」 |
| **任务（Task）** | 任务式项目下的可拆分工作单元，支持子任务 |
| **习惯实例（Habit Occurrence）** | 习惯式项目在某一周期内应执行的一次记录 |
| **里程碑（Milestone）** | 项目中的关键节点，两种项目类型均支持手动设置 |
| **时间记录（Time Entry）** | 与任务或习惯实例关联的耗时记录 |
| **报告（Report）** | 按周/月/年聚合的项目进展与时间统计文档 |

## 3. 功能需求

### 3.1 项目管理

#### 3.1.1 通用能力

- 创建、编辑、归档、删除项目
- 项目字段：名称、描述、图标/颜色、开始日期、目标结束日期（可选）、状态（进行中 / 已归档 / 已完成）
- 项目列表支持按状态、最近活跃、创建时间排序
- 支持手动设置 **Milestone**：
  - 名称、描述、目标日期
  - 完成状态（未开始 / 进行中 / 已完成）
  - 可选关联若干任务或习惯规则
- Milestone 完成时可触发通知（V1 可选，本地通知即可）

#### 3.1.2 任务式项目（Task-based Project）

**适用场景**：软件开发、写作、搬家、考证等可拆分子步骤的项目。

| 能力 | 说明 | V1 优先级 |
|------|------|-----------|
| 子任务树 | 支持多级子任务，任意时刻可新增/编辑/删除/排序 | P0 |
| 任务状态 | 待办 / 进行中 / 已完成 / 已取消 | P0 |
| 任务属性 | 标题、描述、优先级、截止日期、标签 | P0 |
| 任务依赖 | 前置任务完成后才可开始 | P2（V1 不做） |
| 批量操作 | 批量完成、批量移动 | P1 |

**业务规则**：

- 子任务完成度可向上汇总至父任务（全部子任务完成 → 父任务建议标记完成，允许用户覆盖）
- 任务可独立于 Milestone 存在，也可挂载到某个 Milestone 下

#### 3.1.3 习惯养成式项目（Habit-based Project）

**适用场景**：每周健身、每月复盘、每年体检、每日阅读等周期性行为。

| 能力 | 说明 | V1 优先级 |
|------|------|-----------|
| 周期规则 | 支持：每日 / 每周指定日 / 每月指定日 / 每年指定日 | P0 |
| 规则示例 | 「每周一、三、五」「每月 1 日」「每年 6-01」 | P0 |
| 实例生成 | 根据规则自动生成待执行「习惯实例」 | P0 |
| 完成记录 | 标记完成 / 跳过 / 未完成 | P0 |
| 连续统计 | 连续完成天数/周数（Streak） | P1 |
| 灵活调整 | 单次实例可改期（不影响原规则） | P1 |

**周期规则数据结构（概念）**：

```
frequency: daily | weekly | monthly | yearly
daysOfWeek: [1,3,5]        // weekly 时，1=周一
dayOfMonth: 15              // monthly 时
monthAndDay: "06-01"        // yearly 时
```

**业务规则**：

- 过期未完成的实例默认标记为「未完成」，用户可事后补记为「已完成（补录）」
- 习惯式项目同样支持 Milestone，如「连续 12 周完成」或「2026-Q2 养成稳定节奏」

### 3.2 时间记录

#### 3.2.1 记录方式

| 方式 | 描述 | V1 |
|------|------|-----|
| **手动计时** | 用户点击「开始」→ 实时计时 → 点击「停止」保存 | P0 |
| **事后补录** | 任务/习惯完成后，手动填写开始时间、结束时间或总时长 | P0 |
| **快速补录** | 仅填写总时长（如「今天做了 45 分钟」） | P0 |

#### 3.2.2 关联对象

- 时间记录必须关联到：**任务** 或 **习惯实例**
- 可选关联到项目级别（不关联具体任务，V1 可选 P1）

#### 3.2.3 计时器行为

- 同一时刻仅允许 **一个活跃计时器**（跨项目全局唯一）
- 切换任务时提示：停止当前计时 / 取消 / 继续当前
- 计时器在 App 后台/锁屏时继续（移动端需前台服务/实况活动支持，V1 先保证 App 存活时准确）
- 支持暂停与恢复（P1）

#### 3.2.4 时间记录字段

| 字段 | 说明 |
|------|------|
| id | 唯一标识 |
| targetType | task / habit_occurrence |
| targetId | 关联 ID |
| startAt | 开始时间 |
| endAt | 结束时间（计时中为空） |
| durationSeconds | 时长（可由起止计算或手填） |
| note | 备注 |
| source | timer / manual |
| createdAt / updatedAt | 审计字段 |

### 3.3 报告生成（V1.1+，V1 不实现）

> V1 聚焦核心项目管理；报告能力在数据模型中预留 `Report` 实体，UI 与生成逻辑延至 V1.1。

#### 3.3.1 报告类型

| 类型 | 时间范围 | 版本 |
|------|----------|------|
| 周报 | 自然周（可配置周起始日：周一/周日） | V1.1 |
| 年报 | 自然年 | V1.1 |
| 月报 | 自然月 | **不做**（产品决策：无月报需求） |

#### 3.3.2 报告内容（模板化）

**周报 / 年报均包含以下模块（按类型裁剪）**：

1. **概览**
   - 报告周期
   - 活跃项目数
   - 总投入时长
   - 完成任务数 / 习惯完成率

2. **分项目摘要**
   - 项目名称、类型
   - 本周期完成任务列表
   - 本周期时间投入（可下钻到任务级）
   - Milestone 进展（新完成 / 进行中）

3. **习惯式项目专项**
   - 完成率、Streak、漏做次数
   - 按规则周期聚合

4. **时间分布**（V1 简版）
   - 按项目饼图/条形图
   - 按日分布（周报）

5. **备注与反思**（用户可编辑）
   - 自由文本，写入报告快照

#### 3.3.3 报告操作

- 预览（App 内 Markdown 渲染）
- 导出：Markdown、PDF（P1）、纯文本
- 历史报告列表，按类型与时间归档
- 重新生成：基于最新数据覆盖草稿，已「确认归档」的报告不自动覆盖

#### 3.3.4 报告生成策略

- **手动触发**为主（用户选择周期 → 生成）
- V1 不做自动定时推送；可预留「每周一提醒生成周报」本地通知（P2）

### 3.4 思源笔记联动（V1.1+ 可选模块）

> 定位：增强能力，V1 不实现；Rust 层预留 `siyuan` 模块接口。

#### 3.4.1 连接配置

- 思源内核 API 地址（默认 `http://127.0.0.1:6806`）
- API Token 认证
- 连接测试与版本检测（`/api/system/version`）
- 支持局域网 / 远程思源（用户自行保证网络可达）

#### 3.4.2 块（Block）映射配置

用户可为以下场景配置「思源块 ID」：

| 场景 | 行为 |
|------|------|
| 项目级 | 任务状态变更时，更新项目对应块（如进度摘要） |
| 报告级 | 生成周报/月报/年报后，写入或更新指定块 |
| Milestone | 完成时更新指定块 |

配置项：`siyuanBlockId`、`updateMode`（append / replace）、`templateId`（可选）

#### 3.4.3 可用 API（思源内核）

| API | 用途 |
|-----|------|
| `POST /api/block/updateBlock` | 更新块内容（markdown / dom） |
| `POST /api/block/batchUpdateBlock` | 批量原子更新多个块 |
| `POST /api/block/insertBlock` | 插入新块（如追加报告条目） |
| `POST /api/block/getBlockKramdown` | 读取现有内容后再合并更新 |
| `POST /api/block/setBlockAttrs` | 设置块属性（如 custom-* 属性标记状态） |
| `POST /api/template/renderSprig` | 模板渲染（高级用法） |
| `POST /api/export/exportMarkdown` | 导出 Markdown |

**集成注意（来自社区实践）**：

- `updateBlock` 为**覆盖式更新**，更新时需保留原有属性（写入 kramdown 属性语法）
- 内核 API **不支持撤销**，写入前建议本地保留报告副本
- 更新文档块（NodeDocument）会清空子块，应只更新段落/列表块

#### 3.4.4 联动触发点

| 触发事件 | 默认动作 |
|----------|----------|
| 任务标记完成 | 更新项目进度块（可选） |
| 习惯实例完成 | 追加一行完成记录到习惯块（可选） |
| 生成周报/年报 | 将报告 Markdown 写入指定块（replace 或 append） |
| Milestone 完成 | 更新 Milestone 块状态 |

#### 3.4.5 联动范围（V1.1 起）

- **优先**：报告生成 → 写入指定块
- **选做**：项目进度摘要块手动同步
- **不做**：双向同步（以 App 为数据源，思源只读/展示为主）

### 3.5 局域网无主同步（V1 核心）

#### 3.5.1 设计原则

| 原则 | 说明 |
|------|------|
| **离线优先** | 日常使用完全离线，SQLite 为各设备本地真相源 |
| **无主架构** | 无中心服务器、无云账号、不依赖第三方同步服务 |
| **手动触发** | 用户主动点击「同步」，不做后台自动轮询（V1） |
| **局域网限定** | 仅当设备处于同一局域网（如同一 Wi‑Fi）时可发现与同步 |
| **多设备对等** | 手机与电脑地位平等，均为 Peer，无固定「主设备」 |

#### 3.5.2 用户流程

```
设备 A（Mac）                    设备 B（Android）
    │                                │
    │  1. 用户打开「设置 → 同步」      │
    │  2. 点击「开始同步」            │  2. 点击「开始同步」
    │                                │
    └── mDNS 发现局域网内 Spanwork ──┘
              │
    3. 展示可用设备列表（名称 + 平台 + 最后同步时间）
    4. 用户选择目标设备，确认同步
    5. 交换 SQLite changeset，CRDT / LWW 自动合并
    6. 展示结果：成功 / 冲突数 / 需人工确认项（若有）
```

#### 3.5.3 技术要点（Rust 层实现）

| 能力 | 方案 |
|------|------|
| 设备发现 | mDNS / DNS-SD（如 `_spanwork._tcp.local`） |
| 传输 | 局域网 TCP 或 QUIC，TLS 可选（V1 可用配对码鉴权） |
| 冲突合并 | **cr-sqlite** 扩展：表级 CRDT 合并；或 Last-Write-Wins + `updated_at` |
| 设备标识 | 首次启动生成 `device_id`（UUID），用户可编辑设备显示名 |
| 同步粒度 | 增量 changeset（`db_version` 游标），非全库覆盖 |
| 安全 | V1：同一 LAN + 6 位配对码确认；V2 可选 E2E 密钥 |

#### 3.5.4 冲突策略

- **默认**：同记录多设备修改 → 按字段级 LWW（`updated_at` + `device_id` tie-break）
- **删除**：软删除 + tombstone 同步，避免「删了又复活」
- **计时器**：活跃计时器状态不同步（各设备独立）；仅同步已完成的 `TimeEntry`
- **不可自动合并时**：写入「冲突 inbox」，用户选择保留哪一版本（V1 简版可仅日志提示）

#### 3.5.5 V1 同步范围

| 包含 | 不包含 |
|------|--------|
| 项目 / 任务 / 习惯 / Milestone / 时间记录 | 跨公网同步 |
| 手动触发双向 merge | 自动后台同步 |
| 同 LAN 多设备（2 台起） | 多用户协作 |
| 同步历史与最后一次结果摘要 | 实时协作编辑 |

#### 3.5.6 备选方案参考

| 方案 | 说明 | Spanwork 采用 |
|------|------|---------------|
| [cr-sqlite](https://github.com/vlcn-io/cr-sqlite) | SQLite CRDT 扩展，Rust 可集成 | ✅ 首选 |
| [smugglr](https://smugglr.dev/) | LAN UDP 单二进制同步 | 可参考传输层 |
| Syncthing / 文件复制 | 整库文件同步 | ❌ 无行级 merge |
| WebDAV / iCloud | 需第三方 | ❌ 不符合需求 |

## 4. 非功能需求

### 4.1 平台与体验

| 项 | 要求 |
|----|------|
| 响应式布局 | 手机单列、平板/桌面主从视图 |
| 离线优先 | 核心 CRUD 与计时离线可用 |
| 启动性能 | 冷启动 < 3s（中端设备） |
| 数据量 | 支持 5 年+ 个人数据（预估 < 50MB SQLite） |

### 4.2 数据与同步

| 项 | V1 策略 |
|----|---------|
| 存储 | 本地 SQLite（Rust `rusqlite`），单用户 |
| 同步 | **局域网无主同步**：mDNS 发现 + 手动触发 + cr-sqlite changeset 合并 |
| 联网 | App 核心功能**不需要**互联网；思源联动（V1.1）需访问局域网内思源实例 |
| 导出 | JSON 全量导出 + SQLite 文件备份（灾难恢复） |
| 备份 | 手动导出；同步本身即多副本 |

### 4.3 安全与隐私

- 数据默认仅存本地，不上传第三方
- 思源 API Token 存系统密钥链（Keychain / Credential Manager）
- 无账号体系（V1）

### 4.4 可维护性

- Tauri 2 单前端工程 + Rust 后端，覆盖 Android / iOS / macOS / Windows
- 业务逻辑优先放 Rust（DB、同步、计时），UI 层薄，便于 AI 生成 React 代码
- 核心领域模型有单元测试（Rust：习惯规则、changeset merge；TS：表单校验）

## 5. 技术栈调研

> Tauri 2 前端详细对比见：[Tauri2-前端技术栈选型.md](./Tauri2-前端技术栈选型.md)

### 5.1 已确认技术方向：Tauri 2

| 层级 | 选型 |
|------|------|
| **壳** | Tauri 2.x |
| **前端（推荐）** | React + Vite + TypeScript + shadcn/ui + Tailwind |
| **状态** | Zustand + TanStack Query |
| **后端** | Rust：`rusqlite`、`cr-sqlite`、 `mdns-sd`、`tokio` |
| **平台** | Android、iOS、macOS、Windows（鸿蒙 V2+ 另议） |

### 5.2 候选方案对比（历史调研）

| 方案 | 覆盖平台 | AI 辅助开发 | 个人维护 | 鸿蒙 | 综合（当前决策） |
|------|----------|-------------|----------|------|------------------|
| **Tauri 2 + React** | Win/Mac/iOS/Android | **高** | 高（单 UI + Rust） | ❌ V2+ | ⭐⭐⭐⭐⭐ **已选** |
| **Flutter + 鸿蒙社区版** | +鸿蒙 | 中 | 高 | ✅ | ⭐⭐⭐⭐ 鸿蒙备选 |
| **React Native + Tauri** | 双 UI 栈 | 高 | 中 | 社区适配 | ⭐⭐⭐ |

### 5.2 各方案要点

#### Flutter（推荐首选）

**优势**

- 一套 Dart UI 覆盖 **Win / Mac / iOS / Android**，鸿蒙有社区维护的 `3.22.0-ohos` 分支（适配 HarmonyOS NEXT API 16）
- 自绘引擎，跨端 UI 一致性好
- 本地存储成熟：`drift`（SQLite ORM）、`isar` 等
- 个人项目参考：AppFlowy 等开源项目验证可行

**劣势**

- AI 生成 Dart 代码准确率略低于 TypeScript
- 鸿蒙版依赖社区分支，需单独维护构建链（DevEco Studio + flutter_flutter 仓库）
- 桌面端体验弱于原生但可接受

**AI 辅助**：Dart 3.9+ 提供实验性 MCP Server，可与 Cursor 等集成；适合结构化 UI 与业务逻辑生成。

#### React Native + Expo

**优势**

- **AI 辅助开发体验最佳**（TypeScript、npm 生态、Stack Overflow 与训练数据最多）
- Expo EAS 简化 iOS/Android 构建发布
- 鸿蒙有社区 `ohos_react_native` 适配

**劣势**

- **桌面（Win/Mac）无统一方案**，需 Tauri/Electron 另起前端或 React Native macOS/Windows（实验性）
- 五端全覆盖 = 至少两套 UI 工程，**个人维护成本上升**

#### Tauri 2

**优势**

- 桌面体验好、包体小（~3MB 级）
- V2 支持 iOS/Android，前端任意 Web 框架
- Rust 后端适合计时器、文件、同步等系统能力
- **AI 对 TS/React 支持极好**

**劣势**

- 移动端基于 WebView，体验不如 Flutter/RN 原生组件
- **不支持鸿蒙**，需额外独立应用或放弃鸿蒙
- Rust 层调试对个人开发者有门槛

#### 混合方案（备选）

| 组合 | 说明 |
|------|------|
| Flutter 移动端 + Tauri 桌面 | 共享 Dart 业务层困难，需 Rust/Dart 双实现或 FFI，维护成本高 |
| RN 移动 + Tauri 桌面 + 共享 TS 业务包 | AI 友好，但 UI 两套，鸿蒙仍要 RN 适配 |

#### Tauri 2（**当前首选**）

**优势**

- **一套 React 前端**覆盖 Win / Mac / iOS / Android
- Rust 层天然适合 SQLite + LAN P2P 同步（cr-sqlite、mdns）
- 桌面包体小；AI 对 React/TS 辅助效率最高
- 鸿蒙非 V1 目标，不构成选型阻碍

**劣势**

- 移动端 WebView 体验略逊于原生组件（工具类 App 可接受）
- Rust 同步模块需学习，但可让 AI 辅助生成并迭代
- Tauri 移动插件生态仍在完善

### 5.3 数据层方案

| 方案 | 适用 | Spanwork |
|------|------|----------|
| **rusqlite + cr-sqlite** | Tauri Rust 层 | ✅ V1 本地存储 + LAN merge |
| **PowerSync + 后端** | 云同步 | ❌ 需服务器，不符合需求 |
| **整库文件同步** | Syncthing 等 | ❌ 无冲突合并 |

**V1 同步**：cr-sqlite changeset + mDNS 发现 + 用户手动触发 merge。

### 5.4 报告生成技术（V1.1）

- 模板：Rust 或 TS Handlebars / Mustache
- Markdown 渲染：`react-markdown`
- PDF 导出：延后

### 5.5 思源 API 客户端（V1.1）

- Rust `reqwest` 模块，HTTP POST + Token
- 与 UI 解耦，Tauri Command 暴露给前端

### 6.1 整体架构

```
┌─────────────────────────────────────────┐
│  React + Vite + TS + shadcn/ui           │
├─────────────────────────────────────────┤

├─────────────────────────────────────────┤
│  Android · iOS · macOS · Windows         │
└─────────────────────────────────────────┘
```

### 6.2 平台发布顺序

| 阶段 | 平台 | 说明 |
|------|------|------|
| **Alpha** | macOS + Windows | 桌面调试效率最高，先验证 Rust 核心 |
| **Beta** | Android | 移动端主力 |
| **RC** | iOS | 签名与 TestFlight |
| **V2+** | HarmonyOS | 低优先级；评估 Flutter 壳或 ArkUI 简版 |

### 6.3 已确认产品决策

| 决策项 | 结论 |
|--------|------|
| 技术栈 | Tauri 2 + React + Rust |
| 应用名 | **Spanwork**（跨度） |
| V1 范围 | 核心项目管理 + LAN 同步 |
| 月报 | 不做 |
| 报告 / 思源 | V1.1+ |
| 同步 | 局域网无主、手动触发、无第三方 |
| 鸿蒙 | V2+ 低优先级 |

## 10. 风险与待定项

| 风险 | 影响 | 缓解 |
|------|------|------|
| cr-sqlite 与 rusqlite 集成复杂度 | 同步延期 | M0 做 POC；冲突场景单测 |
| Tauri 移动端插件缺口 | 功能受限 | 优先用官方 plugin；计时器放 Rust |
| 局域网 mDNS 在部分路由器失效 | 无法发现设备 | 提供「手动输入 IP:端口」备选 |
| 双设备同时编辑同一记录 | 合并异常 | LWW + 冲突 inbox；避免跨设备同时开计时 |
| AI 生成 Rust 同步代码需审查 | 隐蔽 bug | 同步模块人工 review + 集成测试 |
| iOS 构建与签名曲线 | 发布延期 | 桌面 + Android 先行 |

### 待定项（次要）

1. 同步鉴权：仅配对码 vs 固定设备密钥？
2. V1.1 报告是否先做周报、年报哪个？
3. 鸿蒙 V2：Flutter 壳 vs 放弃？
